# Project Overview

## What is bam?

**bam — Better Audio Mixer** is a per-app audio mixer and router for macOS. It
lets you route any running application to the output device you choose, set
per-app levels, mute/solo individual apps, and watch live meters — all from a
single menu-bar console. An optional Elgato Stream Deck plugin mirrors the same
controls onto physical keys and dials.

The whole thing is built on the CoreAudio **process-tap** API introduced in
macOS 14.4. There is **no kernel extension** and **no SIP changes** required for
the core mixing path: bam captures each app's output with a private process tap,
sums the captured audio at per-source gains inside one hardware-clocked
aggregate device, and renders the mix to the selected output.

## Core Purpose

- Give users a real mixing console for their Mac's audio, at app granularity.
- Do it with modern, sanctioned APIs (process taps) rather than fragile virtual
  drivers or kernel extensions.
- Stay resilient: audio hardware appears and disappears, USB devices
  re-enumerate, and TCC permission grants arrive asynchronously — bam recovers
  from all of these without user intervention.
- Offer a hardware control surface (Stream Deck) that stays perfectly in sync
  with the on-screen console.

## Technology Stack

- **Language**: Swift 6.0 (strict concurrency), plus C for the optional virtual
  audio driver.
- **UI**: SwiftUI, running as an `LSUIElement` menu-bar agent (no Dock icon).
- **Audio**: CoreAudio — process taps (`AudioHardwareCreateProcessTap`),
  aggregate devices, HAL property listeners.
- **Concurrency**: Swift actors (`CoreAudioEngine` is an `actor`), `AsyncStream`
  for live snapshots and events, `swift-atomics` for lock-free RT↔UI cells.
- **Config**: YAML via [Yams](https://github.com/jpsim/Yams), persisted under
  Application Support.
- **Build**: [XcodeGen](https://github.com/yonaskolb/XcodeGen)
  (`project.yml` → `bam.xcodeproj`) + Swift Package Manager (`BamKit`).
- **Control transport**: a local Unix-domain socket speaking NDJSON frames
  (app ↔ Stream Deck plugin).
- **Distribution**: Developer ID signed + notarized `.app`, shipped as a
  Homebrew cask and a `.streamDeckPlugin`.

## Key Features

- Per-app routing and level control, with mute and single global solo.
- A non-deletable **"Everything Else"** remainder source that catches every
  ungrouped app.
- Live per-source and per-mix RMS meters (mono + stereo).
- Per-source pan (a stereo balance law, not equal-power).
- Master fader that scales the whole mix, backed by the OS output volume.
- Automatic, cause-aware **router recovery** (device loss, permission pending,
  no sources running, build failure).
- Stream Deck plugin with three actions (Device, Master, Output Device) and
  three key render styles.
- Click/pop safety: fade-in on aggregate start, tap-mute continuity across
  rebuilds, and a soft output limiter.

## Project Structure

```
App/                       SwiftUI console UI + ConsoleViewModel (menu-bar agent)
BamKit/                     Swift package — all reusable logic
  Sources/BamCore/         Pure model: config, sources, mixes, routing rules
  Sources/AudioEngine/     CoreAudio engine, process taps, router aggregate
  Sources/BamControlKit/   Local control-socket server + wire snapshot types
  Sources/BAMStreamDeck/   Stream Deck plugin executable
  Tests/                   Unit + integration tests per target
StreamDeck/                Stream Deck plugin bundle (manifest, PIs, layouts)
BAMDriver/                 Optional virtual audio driver (C, BlackHole-derived)
Casks/bam.rb               Homebrew cask
project.yml                XcodeGen project definition
Makefile                   build / install / test / release helpers
```

## Getting Started

Install via Homebrew:

```sh
brew tap lkshrk/tap
brew install --cask lkshrk/tap/bam
```

Or grab `bam.zip` from the latest GitHub release, unzip, and drag `bam.app` to
`/Applications`. On first launch bam prompts for **audio-capture permission**,
which it needs to meter and route other apps' audio.

Build from source (requires XcodeGen and Xcode 16+):

```sh
xcodegen generate
xcodebuild -project bam.xcodeproj -scheme bam -configuration Release \
  -derivedDataPath .build build
open ".build/Build/Products/Release/bam.app"
```

## Architecture Summary

bam is a **layered** application with a clean split between a pure model and the
side-effecting audio layer:

1. **BamCore** holds the entire routing model as value types (`BamConfig`,
   `Source`, `Mix`, `Send`) and the pure rules that validate and normalize it.
   It knows nothing about CoreAudio.
2. **AudioEngine** is an `actor` (`CoreAudioEngine`) that turns a `BamConfig`
   into live process taps and one summing aggregate device, and streams back
   meter snapshots and health/recovery events.
3. **App** hosts the SwiftUI console and `ConsoleViewModel`, which owns the
   config, drives the engine, and reacts to its event streams.
4. **BamControlKit** exposes the running model over a local socket so the
   **BAMStreamDeck** plugin (a separate process) can mirror and drive it.

The next document, *Architecture Overview*, details these boundaries with C4
diagrams. See *Workflow Overview* for the runtime sequences (routing edits,
recovery, remote control), and the *Deep Dive* folder for per-subsystem detail.
